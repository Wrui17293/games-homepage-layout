# 首页低保真布局决策规范

执行本规范时，先读取 `references/rules.md`，再按本文完成页面骨架、槽位状态和组合关系判断。

## 目标
把 PRD 中不稳定的首页模块，转换为可执行的页面骨架、槽位状态和组合关系。

不要先按固定顺序堆模块。必须依次完成：

`选择页面骨架 → 判断槽位是否出现 → 定义槽位关系 → 排布首屏 → 检查遮挡`

## 标准名称

其他文件必须使用本文件中的名称，不创建同义名称：

- `头图氛围区`，不写“背景氛围”。
- `核心玩法区`，不写“核心游戏区”。
- `辅按钮`，不写“副按钮”。
- `资源位`，不写“资源位卡片”。
- `Feeds`，不写“商品 Feeds”。
- 账户、等级、体力、次数、资产等信息统一归入 `常驻功能信息区`。

## 一、页面骨架

核心玩法区高度按“首屏可用高度”计算：

`首屏可用高度 = 首屏高度 − 顶部通栏实际占用高度 − 底部固定区域实际占用高度`

以下百分比为推荐范围，不是固定值。最终以玩法内容完整、主要操作可见、无裁切为准。

### A. 全屏沉浸型
适用：单一核心玩法、强场景感、首页即游戏主界面。

- 核心玩法区建议占首屏可用高度的 `60%–100%`。
- 核心玩法区占首屏主体。
- 头图氛围区与核心玩法区允许融合。
- 主按钮出现时，靠近玩法结果或下一步动作。
- Feeds 出现时，放在下拉区域，不压缩核心玩法区。

### B. 半屏效率型
适用：上半屏完成玩法或转化，下半屏按 PRD 承接任务、奖励、资源位或 Feeds。

- 核心玩法区建议占首屏可用高度的 `40%–60%`。
- 首屏上部通常为头图氛围区与核心玩法区。
- 主按钮、进度或奖励反馈出现时，优先靠近核心玩法区。
- 首屏下部可按 PRD 出现任务、奖励、资源位或 Feeds。
- 核心玩法区不得为了露出 Feeds 被过度压缩。

### C. 三分屏内容型
适用：活动信息、核心操作、内容承接同等重要。

- 核心玩法区建议占首屏可用高度的 `25%–45%`。
- 第一段：通常放头图氛围区，并按 PRD 判断是否需要主标题、副标题或活动利益点。
- 第二段：以核心玩法区为主，状态信息与主辅按钮按 PRD 选择性出现。
- 第三段：按 PRD 承接任务、记录、资源位或 Feeds。
- 三段可以在同一长页面内自然滚动，不要求每段等高。

## 二、A/B/C 共用槽位规则

以下模块不归属于某一种骨架，统一适用于 A、B、C。

### 顶部通栏

- 所有首页必须出现，槽位状态固定为 `必选`。
- 组合关系固定为 `叠加`，覆盖在页面背景、头图氛围区或核心玩法区上方，不参与普通纵向顺序。
- 顶部通栏虽然采用叠加关系，但其安全避让区需要从首屏可用高度中扣除。
- 顶部通栏内部的返回、页面名称、奖品入口和更多入口按组件既有结构使用，不在首页布局阶段重新排列。
- 顶部通栏的素材链接、尺寸和可修改范围统一引用 `games-component-specs`。

除顶部通栏外，其余共用槽位均需根据 PRD 判断是否出现。

### 核心玩法区

核心玩法使用两层信息：

1. `玩法目标`：来自 `games-prdbrief`，例如抽奖、休闲、收集、养成、解谜，用于理解业务目的。
2. `交互结构`：在本阶段根据用户动作、操作对象、过程和结果判断，用于指导低保真表达。

交互结构分为：

- `格子选择型`：九宫格、翻牌、棋盘、拼图等。重点体现格子数量、排列、选择和状态变化。
- `器械操作型`：转盘、刮刮乐、摇奖机等。重点保留决定操作方式的圆盘、刮层、摇奖口等关键结构。
- `单主体操作型`：IP、大红包、盲盒、扭蛋、宝箱等。适用于主体主要承担点击、开启、领取或触发操作的情况。
- `连续操控型`：跳一跳、接红包、打地鼠、向前冲等。重点体现操作区域、运动方向、目标和即时反馈。
- `场景探索型`：地图、房间、农场、店铺等。重点体现场景范围、点击热点、路径和状态信息位置。
- `收集成长型`：集卡、种树、养宠物、经营店铺等。重点体现成长主体、资源、进度和阶段状态。
- `题目决策型`：答题、排序、找不同、你画我猜等。重点体现题面、选项或作答区域和结果反馈。

判断规则：

- 按交互结构分类，不按最终视觉外形分类。
- 同一个玩法允许使用一个主要结构加一个辅助结构，例如“养成 + 农场”可判断为 `收集成长型 + 场景探索型`。
- 主体内部结构会影响用户理解操作方式时，优先判断为 `器械操作型`；仅承担点击、开启或领取时，判断为 `单主体操作型`。
- 交互结构只决定低保真如何表达，不决定最终视觉风格。

### 常驻功能信息区

- A、B、C 三种骨架均可出现。
- 组合关系固定为 `叠加`，不得设置为独立、包含、贴附、悬浮或融合。
- 常驻功能信息区叠加在头图氛围区或核心玩法区上方，不参与普通纵向顺序，也不计入核心玩法区高度占比。
- 允许位于核心玩法区顶部、左侧或右侧。
- 多个侧边入口应沿屏幕边缘成组排列，不得分散遮挡核心角色或主要操作区域。
- 常驻入口与核心玩法区发生视觉重叠时，必须声明点击热区和安全避让区。
- 首屏空间不足时，优先隐藏、折叠或收入“更多”低优先级入口，不得压缩核心玩法区。
- 单侧入口超过 3 个时，应考虑折叠、轮换展示或使用二级入口。
- Icon、蒙层透明度、圆角、字号、间距和组件状态统一引用 `games-component-specs`，本文件不重复定义。

### 头图氛围区

- A、B、C 三种骨架均可出现，不与某一种骨架绑定。
- 默认保留头图氛围区，用于承载活动主题、利益点或场景氛围；主标题和副标题按 PRD 选择性出现。
- 头图氛围区可以独立存在，也可以与核心玩法区、主标题融合。
- 当首屏使用大面积单坑位或双坑位，且坑位本身已经占据主要展示面积时，可以不设置独立头图氛围区。
- 省略头图氛围区时，活动名称、必要说明和核心操作仍需有明确承载位置，不得直接丢失。
- 不得为了同时容纳大坑位和头图氛围区而过度压缩坑位内容。
- 三坑位及以上、坑位面积较小或需要强化活动主题时，默认保留头图氛围区。

### 主标题与副标题

- A、B、C 三种骨架均可能出现，也均可能不出现。
- 主标题和副标题必须分别判断，不得因为存在头图氛围区就自动生成。
- 当顶部通栏、核心玩法画面或大坑位已经清楚表达活动名称和核心利益点时，主标题可以不出现。
- 当主标题已经足以说明活动内容，且没有必要补充玩法、时间或利益点时，副标题可以不出现。
- 主标题和副标题出现时，可以位于头图氛围区内，也可以与核心玩法区融合。
- 标题省略后，不得影响用户理解当前活动、核心利益或主要操作。

### 资源位

- A、B、C 三种骨架均可能出现，也均可能不出现。
- 是否出现完全依据 PRD，不得因为页面下方存在剩余空间而自动补充。
- 当 PRD 有跨活动导流、频道入口、任务入口或业务推荐时，可设置资源位。
- 当首页只需要完成核心玩法，或资源位会分散核心转化时，应标记为“不出现”。
- 资源位出现时，可位于核心玩法之后或 Feeds 之前，但不得压缩核心玩法区。
- 资源位的 Icon、卡片尺寸和组件形式统一引用 `games-component-specs`。

### 进度激励区

- A、B、C 三种骨架均可能出现，也均可能不出现。
- 仅当 PRD 存在收集、助力、累计任务、阶段解锁或进度兑换机制时使用。
- 用于展示“当前完成到哪里、下一阶段可获得什么、接下来需要做什么”，不替代核心玩法区。
- 可分为 `阶段奖励进度` 和 `目标任务进度`：
  - 阶段奖励进度：展示多个奖励节点、节点状态、最终奖励和截止时间。
  - 目标任务进度：展示当前值、目标值、阶段节点、奖励说明和行动按钮。
- 默认作为独立模块放在核心玩法区之后；当进度是玩法的必要组成部分时，可与核心玩法区融合。
- 若模块包含行动按钮，最多突出一个主操作，其余操作降为辅操作。
- 不得仅因页面存在任务或奖励，就自动生成进度激励区；没有明确进度机制时标记为“不出现”。
- 进度条、奖励节点、倒计时和按钮样式统一引用 `games-component-specs`。

## 三、槽位出现状态

每个槽位必须标记一种状态：

- `必选`：缺失会导致核心流程无法完成。
- `条件出现`：满足业务、状态或用户条件时出现。
- `可选`：用于增强理解、氛围或转化。
- `不出现`：本页明确不使用。

不得因为槽位清单中存在某模块，就默认把它画进首页。

## 四、槽位组合关系

每两个强关联槽位必须选择一种关系：

- `独立`：各自占据文档流空间，边界清晰。
- `包含`：一个槽位位于另一个槽位内部，例如标题包含在头图氛围区。
- `叠加`：视觉层发生覆盖，但必须声明上层、下层和安全区。
- `贴附`：模块贴在目标模块边缘，例如主按钮贴附在核心玩法区底部。
- `悬浮`：脱离文档流固定在屏幕边缘，例如顶部通栏、侧边功能入口。
- `融合`：两个槽位共同构成一个主模块，不再保留各自独立卡片，例如关卡区与核心玩法区。

禁止只写“可以重叠”。使用叠加、贴附或悬浮时，必须写明：

- 基准槽位
- 对齐边
- 覆盖方向
- 最大覆盖范围
- 点击热区
- 安全避让区

## 五、独立槽位间距规则

- 本规则只约束页面边缘与独立槽位之间、独立槽位与独立槽位之间的外部间距。
- 所有外部间距统一使用 `3px` 网格，取值必须为 3 的倍数。
- 允许值为：`3、6、9、12……99px`，即 `3 × n`，其中 `n = 1–33`。
- 页面左右安全边距默认使用 `30px`；有明确布局需求时可以改用其他允许值，但左右应保持一致。
- 相邻独立槽位的纵向间距必须从允许值中选择，不得使用 `10px、16px、20px` 等非 3 倍数。
- 同组并列独立槽位的横向间距也必须从允许值中选择。
- 独立槽位不得使用 `0px` 间距；若两个槽位无间距，应改为“包含”“叠加”“贴附”或“融合”关系。
- 单处间距不得超过 `99px`；需要更大留白时，应通过调整模块高度、拆分页面段落或重新选择页面骨架处理。
- 带溢出装饰时，最终间距仍需落在 3px 网格上：先计算安全间距，再向上取整到最近的 3 的倍数。
- 组件内部的 Icon、文字、按钮等间距不由本文件定义，统一引用 `games-component-specs`。

推荐常用档位：

- `6–12px`：紧密关联。
- `15–24px`：同组内容。
- `30px`：页面左右边距或常规模块间距。
- `36–60px`：不同内容区之间。
- `63–99px`：强分段留白，谨慎使用。

## 七、选择原则

出现冲突时按以下顺序决策：

`核心玩法可理解 → 核心操作可完成 → 状态信息可见 → 营销利益点清晰 → 内容丰富度`

无法从 PRD 判断骨架或组合关系时，优先按“核心玩法可理解、核心操作可完成”的原则选择一个最稳妥方案，并在内部记录设计假设，不在低保真页面中增加说明文字。只有缺失信息会导致无法形成有效低保真稿时，才向用户提问。
